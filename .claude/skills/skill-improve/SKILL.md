## 概要

スキルのフィードバックファイルと使用ログを分析し、SKILL.md の改善案を提示・適用する。

## 動作モード

| モード | トリガー | 動作 |
|--------|----------|------|
| 自動スキャン | `$ARGUMENTS` 未指定 | 全スキルをスキャンし、フィードバックのあるスキルを自動検出・一括改善 |
| 指定スキル | `$ARGUMENTS` にスキル名指定 | 指定スキルのみ改善（従来動作） |

## 手順

### 1. 対象スキルの決定

**`$ARGUMENTS` にスキル名が指定されている場合**: そのスキルを対象にステップ 2 へ。

**`$ARGUMENTS` が未指定の場合**: 自動スキャンモードで動作する。

1. `{cwd}/.claude/skills/*/feedback/` を走査し、`applied/` を除く `.md` ファイルを持つスキルを列挙する
2. 各スキルについて、シグナルを持つフィードバック（`has_signals: true` または手動作成）の件数をカウントする
3. 結果をサマリー表示する:

```
## フィードバックスキャン結果

| スキル | FB件数 | 実質件数 | 対象 |
|--------|--------|---------|------|
| tdd | 5 | 3 (signal) | ✅ |
| ts-implement | 2 | 0 (signal) | - |
| react-implement | 4 | 2 (signal) | ✅ |
```

4. 実質件数が **1件以上**のスキルを改善対象とする
5. 対象が0件の場合、「改善対象のスキルはありません」と伝えて終了する
6. 対象がある場合、一覧を示して確認する:

> 上記のスキルを順番に改善します。よろしいですか？
> - `all`: 全て実行
> - `tdd react-implement`（スキル名をスペース区切り）: 指定のみ実行
> - `cancel`: キャンセル

7. 承認後、対象スキルごとにステップ 2〜8 を繰り返す

### 2. ファイルの読み込み

以下のファイルを読み込む。

**SKILL.md の読み込み先（優先順）**:
1. `{cwd}/.claude/skills/{target}/SKILL.md`（プロジェクト固有スキルが存在する場合）
2. `~/.claude/skills/{target}/SKILL.md`（グローバルスキル）

**フィードバックと tracker の読み込み先**（作業中プロジェクトの蓄積データ）:
- `{cwd}/.claude/skills/{target}/feedback/` 配下の全 `.md` ファイル（`applied/` サブディレクトリは除く）

> **auto_generated ファイルの扱い**: `auto_generated: true` フィールドを持つファイルは Stop フックによる自動収集です。`has_signals: true` のファイルには不満シグナルやSkill呼び出し後のユーザー応答が含まれるため、分析対象とします。`has_signals: false` のファイルはメタデータのみなので分析対象から除外してください。
- `{cwd}/.claude/skills/_tracker/usage.jsonl`（使用頻度・コンテキスト）
- `{cwd}/.claude/skills/_tracker/insights.jsonl`（Insight ログ。`session_id` で `usage.jsonl` と照合して特定スキル使用時の Insight を特定できる）

> ここでの `{cwd}` はスキル実行時の作業ディレクトリ（プロジェクトルート）を指す。

フィードバックファイルが0件の場合は以下を伝えてスキップする（自動スキャンモードでは次のスキルへ進む）。

> {target} のフィードバックがありません。

### 3. 分析

以下の観点でフィードバックを分析する。

- 言及されたセクションを集計する（複数件のフィードバックで同じセクションが挙げられていれば優先度が高い）
- 繰り返し報告される問題パターンを特定する
- 使用ログの `cwd` や `session_id` から使用コンテキストを把握し、現在の SKILL.md で不足している内容を特定する
- `insights.jsonl` の内容を参照し、当該スキル使用セッション（`session_id` が一致するもの）で生成された Insight から「スキル使用時に補足説明が必要だった箇所」を特定する

### 4. 改善案の提示

分析結果を元に、以下のフォーマットで改善案を提示する。

```
## {skill-name} の改善案

### 変更 1: {セクション名}
**現在の内容**: （該当箇所の抜粋）
**提案する変更**: （変更後の内容）
**変更理由**: （フィードバック N件が根拠。具体的な問題パターンを説明）

### 変更 2: {セクション名}
...
```

### 5. ユーザー承認の取得

提案後、以下の選択肢を提示してユーザーの判断を待つ。

> 上記の変更をどのように適用しますか？
> - `all`: 全て適用
> - `1 3`（番号を半角スペース区切り）: 指定した番号のみ適用
> - `skip`: このスキルをスキップ（自動スキャンモード時、次のスキルへ）
> - `cancel`: 全体をキャンセル

### 6. SKILL.md への適用

承認された変更を SKILL.md に適用する。

- frontmatter（`---` で囲まれた YAML ブロック）は保持する
- `name` と `description` フィールドは変更しない
- 既存の内容との整合性を保ちながら変更を加える

### 7. フィードバックファイルの移動

適用に使用したフィードバックファイルを `feedback/applied/` ディレクトリに移動する。

- 移動先ディレクトリが存在しない場合は作成する
- ファイル名は変更しない

### 8. 完了サマリーの表示

以下の内容を含むサマリーを表示する。

- 適用した変更の一覧（変更番号とセクション名）
- 移動したフィードバックファイルの一覧
- 更新後の SKILL.md のパス

自動スキャンモードの場合、全スキル完了後に全体サマリーも表示する:

```
## 全体サマリー

| スキル | 変更数 | 処理FB数 |
|--------|--------|---------|
| tdd | 2 | 3 |
| react-implement | 1 | 2 |
| **合計** | **3** | **5** |
```
