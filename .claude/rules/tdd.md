# TDD ルール

## 原則

**コードを実装する際は、必ずテストを先に書くこと。**

プロダクションコードの実装より前にテストが存在しない場合、その実装は TDD に違反している。

## Red-Green-Refactor サイクル

すべての実装はこのサイクルに従うこと:

1. **Red**: 失敗するテストを書く
2. **Green**: テストが通る最小限のコードを書く
3. **Refactor**: テストを通したままコードを整理する

このサイクルを1テストケースずつ繰り返す。複数のテストをまとめて書いてから実装するのは禁止。

## 禁止事項

- プロダクションコードを先に書き、後からテストを追加する
- テストなしでプロダクションコードを実装する
- 「シンプルな変更だからテスト不要」と自己判断してテストを省略する
- すべてのテストケースをまとめて書いてから実装する（1サイクルずつ進めること）

## テストケースの設計順序

1. 正常系（ハッピーパス）
2. 境界値
3. エラーケース・異常系
