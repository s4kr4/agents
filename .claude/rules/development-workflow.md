# 開発ワークフロー

エージェント間の責任範囲と開発フローを定義します。

## 原則

- エージェント一覧: `~/.claude/agents/INDEX.md`
- **コーディングおよびそれに類する作業はエージェント経由で行うこと**
  - 調査・計画・実装・検証のいずれのフェーズも、自ら直接実行せず適切なエージェントに委任する
  - 対象: コードの新規実装・追加、バグ修正、リファクタリング、設定ファイルの変更、テストの作成・修正
  - 非対象（直接対応してよい）: 説明・解説・Q&A・ドキュメントの読み解きなど、ファイルへの書き込みを伴わない作業
- **使用するエージェントが不明な場合は、自己判断せずユーザーに確認する**
- **タスクの複雑さに関わらず、自己判断でフェーズをスキップしない**
  - 「シンプルな変更」と思っても、最低限 `@code-investigator` での調査を行う

```
┌─────────────────────────────────────────────────────────────┐
│                    Development Workflow                      │
└─────────────────────────────────────────────────────────────┘

                  │

                  │
Phase 2: Planning (Plan Mode) ※単純な変更はスキップ可

│  ────────────────────────────────────    │

│  ✅ タスクの分解と優先順位付け          │
│  ✅ トレードオフの整理                  │
│  ✅ ユーザー承認の取得                  │

                  │
                  ▼ 計画承認
                  │
Phase 3: Implementation
┌──────────────────────────────────────────┐

                  │

                  │

        ❌ 問題 → 実装エージェントへフィードバック
```

## Usage Examples

### フェーズ選択ガイド

| 変更の複雑さ                           | 調査 | 計画 | 実装 | 検証 |
| -------------------------------------- | ---- | ---- | ---- | ---- |
| 複雑（多ファイル、アーキテクチャ変更） | ✅   | ✅   | ✅   | ✅   |
| 中程度（複数ファイル、明確なパターン） | ✅   | ✅   | ✅   | ✅   |
| 単純（1-2ファイル、軽微な変更）        | ✅   | ❌   | ✅   | ✅   |

**実装フェーズは変更の複雑さに関わらず常に実装エージェントに委任すること。**
自ら直接 Edit/Write でコードを実装することは禁止。

### 通常の開発パターン

#### パターン1: 完全な4フェーズ実行（複雑な変更）

```
@code-investigator でコードベース調査
   ↓
@code-planner で実装計画策定・承認
   ↓
@web-api-implementer または @web-ui-implementer で実装
   ↓
@code-safety-inspector で検証
```

#### パターン2: 計画フェーズ込み（中程度の変更）

```
@code-planner で計画策定（調査も含む）
   ↓
@web-api-implementer または @web-ui-implementer で実装
   ↓
@code-safety-inspector で検証
```

#### パターン3: 計画スキップ（単純な変更）

```
@code-investigator でコードベース調査
   ↓
@web-api-implementer または @web-ui-implementer で実装
   ↓
@code-safety-inspector で検証
```

### 問題が見つかった場合

```
@code-safety-inspector: エラー検出
   ↓
レポート作成（エラー箇所と修正案を提示）
   ↓
@web-api-implementer または @web-ui-implementer: 修正実装
   ↓
@code-safety-inspector: 再検証
```

## ハーネス反復の原則

### 基本原則

> 「ハーネスの各コンポーネントは、モデルが単独ではできないことへの仮定を表す。その仮定は定期的に検証する価値がある。」

ハーネスの設計空間は、モデルが改善しても縮小しない——**移動する**。

### モデル更新時のレビューチェックリスト

新しいモデルへ切り替える際、以下を確認する：

- [ ] 各エージェントフェーズは今のモデルでも必要か？
- [ ] スキップ可能になったフェーズはないか？（不要なスキャフォールドの削除）
- [ ] 新しいモデルの能力で達成できるようになった、これまで不可能だったことはないか？
- [ ] コンテキスト管理戦略は現在のモデルに合っているか？

### フェーズ削減の判断基準

あるフェーズを削減・統合する場合：

1. そのフェーズをスキップして実験する
2. 出力品質を比較する
3. 品質劣化がなければ削減を採用
4. 劣化があれば元に戻す（一度に1フェーズずつ変更する）

## コンテキスト管理ガイドライン（長時間タスク）

### Context Anxiety（コンテキスト不安）

モデルはコンテキスト窓が埋まってくると、タスクを早期に完了しようとする傾向がある（context anxiety）。

**対策**：

- 実装エージェントは長時間タスクで進捗が止まったらコンテキストリセットを検討する
- コンテキストリセット時は、次のエージェントが作業を引き継げる**ハンドオフアーティファクト**をファイルに書き出す

### ハンドオフアーティファクトの構成

セッション間でコンテキストを引き継ぐ場合、以下を含むファイルを作成する：

```markdown
# ハンドオフ: [タスク名]

## 完了済み
- [実装済みの機能・ファイル]

## 次のステップ
- [残りのタスク（優先順）]

## 重要な決定事項
- [実装中に行った設計上の決定とその理由]

## 既知の問題
- [未解決の問題・注意が必要な箇所]
```

### Compaction vs Context Reset

| 手法 | 採用条件 |
|------|---------|
| **Compaction（自動圧縮）** | モデルが長時間タスクで安定している場合 |
| **Context Reset（手動リセット）** | タスクが止まる・早期完了しようとする兆候がある場合 |
